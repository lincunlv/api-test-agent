# A1 业务逻辑梳理

## 输出目标

基于代码 diff、接口链路和现有结构化证据说明本次改动涉及的业务目的、主流程、关键分支、外部依赖和待确认项，并补充测试视角下的关键验证点，帮助后续 A2、A3、A4 产出更聚焦。

## 固定输出结构

1. 业务目标
2. 测试视角摘要
3. 主流程
4. 关键分支
5. 外部依赖
6. 测试关注点
7. Mermaid 流程图
8. 待确认项

## 分析要求

1. 优先根据 git diff 识别本次改动涉及的入口、服务、数据访问和状态流转，再用“先入口、后服务、再数据访问”的方式梳理链路。
2. 识别参数校验、状态判断、幂等校验、事务边界、外部调用和持久化动作。
3. “测试视角摘要”需用 3 到 5 条说明本次改动最值得优先验证的主路径、关键分支和高风险依赖。
4. “主流程”必须写清每一步的输入条件、结果产出和测试关注点，避免只写开发视角描述。
5. “关键分支”优先覆盖失败返回、状态不满足、重复请求、外部依赖失败、空结果和异常处理。
6. “外部依赖”要说明失败后是否影响主流程、是否有降级/兜底、测试时看什么日志或数据。
7. 若 diff 无法稳定定位明确接口，应退化为代码层链路分析，并在“待确认项”中说明未识别到的入口或接口。
8. 对不确定逻辑必须显式标记。
9. Mermaid 图必须体现关键判断节点、失败路径、外部依赖调用点，以及与本次 diff 直接相关的关键改动点。
