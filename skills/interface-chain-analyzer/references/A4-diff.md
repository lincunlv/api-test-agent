# A4 Diff 场景级测试脚本生成

## 输出目标

根据 git diff、受影响代码和接口链路，识别本次改动直接影响和间接受影响的接口、业务场景与状态流转，并优先生成可直接落地的场景级测试脚本草稿。

输出重点不是重写一份全量测试清单，而是基于“这次改动”明确：

1. 哪些业务场景需要补脚本。
2. 每个场景由哪些接口串联构成。
3. 每个场景脚本为什么要补。
4. 脚本应如何构造请求、串联步骤、断言结果和观测关键点。
5. 哪些旧能力需要补充跨接口回归脚本。

## 固定输出结构

1. 变更摘要
2. 测试脚本生成判断
3. 关联接口识别
4. 受影响场景识别
5. 增量测试脚本
6. 回归补充脚本
7. 执行顺序建议
8. 待确认项

## 分析要求

1. 优先识别本次 diff 直接影响的 Controller、Service、Repository、DTO、异常处理、状态流转、方法解析逻辑和返回结构。
2. 必须区分直接影响接口和间接受影响接口，并说明关联依据。
3. 变更摘要必须指出本次变更属于哪类影响：请求参数、响应字段、业务分支、状态流转、异常处理、依赖调用、持久化、鉴权、幂等、日志审计。
4. `测试脚本生成判断` 至少回答以下问题：
	- 是否需要新增接口测试脚本。
	- 是否需要补充旧能力回归脚本。
	- 是否存在高风险误伤面需要优先脚本化验证。
5. 必须尽量把接口影响提升为“业务场景影响”表达，优先识别主流程、逆向流程、状态流转、依赖失败、幂等重复、权限校验和回归场景。
6. `受影响场景识别` 不能只列接口名，必须说明场景入口、接口链、共享业务对象或主键、字段传递提示、状态提示、触发条件和场景依据。场景依据应来自 diff、链路数据、共享业务对象、主键传递、字段绑定、状态流转或关联接口证据。
7. 增量测试脚本必须按“新增验证点 / 变更触发点 / 需要重点回归的旧能力”组织，而不是重写一份全量接口测试清单。
8. 每条脚本都要说明触发原因，原因必须来自 diff、受影响方法、受影响场景或关联接口证据。
9. `请求数据/输入` 必须尽量体现本次变更触发条件，包含请求方法、路径、关键参数、关键 Header、共享业务主键或前置状态，避免只写笼统步骤。
10. `执行步骤/脚本骨架` 必须给出可直接落地改写的脚本草稿，至少体现请求构造、跨接口调用顺序、上一步响应字段到下一步请求字段的绑定关系和关键断言位置。脚本形式优先贴近接口测试场景，可使用 curl、HTTP 请求片段、伪代码或仓库既有风格。
11. `断言` 必须体现脚本校验点，例如 HTTP 状态码、响应字段、错误码、状态流转结果、持久化结果或异常信息。
12. `核心观测点` 必须体现测试落点，例如返回字段、日志、changedMethods、relatedInterfaces、库表、缓存、任务状态、业务状态或异常信息。
13. 回归补充脚本必须区分：`必回归`、`建议回归`、`暂不回归`。
14. 对无法从 diff 直接确认的部分标记为待确认，禁止编造。
15. 若存在工具输出的 `relatedInterfaces`，优先以该结构化结果作为关联接口证据来源。
16. 若 diff 无法定位到明确接口，必须输出“未识别到明确接口，仅能给出代码层回归脚本建议”；若能识别接口但不能稳定推导完整场景，也必须显式标记“场景链待确认”。
