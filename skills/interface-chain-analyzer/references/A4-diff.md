# A4 Diff 关联接口用例生成

## 输出目标

根据 git diff、受影响代码和接口链路，识别关联接口并生成真正可执行的增量测试与回归测试方案，优先帮助团队验证“这次改动会不会破坏已有能力、需要补测哪里、先测什么”。

## 固定输出结构

1. 变更摘要
2. 测试影响判断
3. 关联接口识别
4. 增量测试用例
5. 回归建议
6. 执行顺序建议
7. 待确认项

## 分析要求

1. 优先识别本次 diff 直接影响的 Controller、Service、Repository、DTO、异常处理、状态流转、方法解析逻辑和返回结构。
2. 必须区分直接影响接口和间接受影响接口，并说明关联依据。
3. 变更摘要必须指出本次变更属于哪类影响：请求参数、响应字段、业务分支、状态流转、异常处理、依赖调用、持久化、鉴权、幂等、日志审计。
4. `测试影响判断` 至少回答以下问题：
	- 是否需要新增增量测试。
	- 是否需要重点回归旧能力。
	- 是否存在高风险误伤面。
5. 增量测试用例必须按“新增验证点 / 需要重点回归的旧能力”组织，而不是重写一份全量接口测试清单。
6. 每条用例都要说明触发原因，原因必须来自 diff、受影响方法或关联接口证据。
7. `测试数据/输入` 必须尽量体现本次变更触发条件，避免只写笼统步骤。
8. `核心观测点` 必须体现测试落点，例如返回字段、日志、changedMethods、relatedInterfaces、库表、缓存或异常信息。
9. 回归建议必须区分：`必回归`、`建议回归`、`暂不回归`。
10. 对无法从 diff 直接确认的部分标记为待确认，禁止编造。
11. 若存在工具输出的 `relatedInterfaces`，优先以该结构化结果作为关联接口证据来源。
12. 若 diff 无法定位到明确接口，必须输出“未识别到明确接口，仅能给出代码层回归建议”。
