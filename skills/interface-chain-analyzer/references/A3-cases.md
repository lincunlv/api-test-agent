# A3 测试用例生成

## 输出目标

根据代码 diff、链路主流程、分支条件、异常处理和外部依赖生成测试用例，确保覆盖主路径、边界条件、高风险点以及本次最值得优先执行的验证场景。

## 固定输出结构

1. 覆盖策略摘要
2. 测试用例表格
3. 执行顺序建议
4. 待确认项

## 分析要求

1. 至少覆盖本次改动直接触发的正常流程、参数边界、异常分支、状态流转、幂等/重复提交和外部依赖失败场景。
2. `用例类型` 优先使用：主流程、分支、边界、异常、依赖失败、状态流转、回归。
3. `测试数据/输入` 要尽量写清关键参数、数据状态或 mock 条件，避免只写抽象描述。
4. `核心观测点` 必须说明测试时重点看什么，例如返回字段、错误码、日志、数据库、缓存或下游调用结果。
5. 优先级需体现业务风险和核心路径重要性，统一使用 `P0 / P1 / P2`。
6. `执行顺序建议` 应优先帮助测试先跑冒烟、再跑高风险、最后补边界和低优先级场景。
7. 若存在 `relatedInterfaces` 或 `scenarioCandidates`，应优先据此组织跨接口或场景级用例。
8. 若 diff 无法定位到明确接口，需输出代码层回归用例并标记待确认的接口或场景边界。
9. 测试步骤和预期结果应可直接转为手工或自动化用例。
