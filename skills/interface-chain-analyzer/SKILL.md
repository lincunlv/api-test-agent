# interface-chain-analyzer

## 1. 目标

本 Skill 用于消费接口链路数据，并在固定结构下生成 A1-A5 五类分析结果。

所有输出默认服务于“本次接口分析或代码变更验证”，目标是让测试、开发和质量同学可以直接拿结果做增量测试、回归测试、风险核对和联调准备，避免模型自由发挥造成的格式漂移、信息失焦和覆盖不稳定。

## 2. 意图路由规则

当用户请求符合以下意图时，必须映射到指定节点：

| 节点 | 意图 | 典型关键词 |
| --- | --- | --- |
| A1 | 业务逻辑梳理 | 业务流程、逻辑梳理、链路解读、流程图 |
| A2 | Bug 风险分析 | 风险分析、异常点、Bug、隐患、边界问题 |
| A3 | 测试用例生成 | 测试用例、测试点、覆盖设计、场景设计 |
| A4 | Diff 场景级测试脚本生成 | git diff、代码变更、变更影响、场景测试、接口测试脚本、增量测试、回归脚本、关联接口、状态流转 |
| A5 | API 文档生成 | API 文档、接口说明、请求响应、文档补齐 |

## 3. 强制执行顺序

1. 先读取本文件，确定用户意图与节点映射。
2. 再读取目标节点对应的 reference 文件。
3. 严格按照 reference 文件中的输出结构、章节顺序和表头生成结果。
4. 若输入中包含 git diff、changedFiles、changedMethods、relatedInterfaces、source-context 等结构化证据，必须优先消费这些证据。
5. 若信息不足以支撑确定性结论，必须输出“待确认项”或“无法判断”，禁止编造。

## 4. 共通分析原则

1. 优先基于链路数据、git diff、方法源码和明确证据下结论。
2. 输出必须围绕“如何测试这次接口或变更”组织，优先回答以下问题：
	- 变更或链路影响了什么。
	- 哪些接口、分支、状态或依赖需要优先验证。
	- 需要准备什么数据、观察什么结果、回归什么旧能力。
3. 只有在必要时才补充方法源码查询，避免无关扩写。
4. 结论必须尽量给出依据、触发条件、影响范围、验证方式或观测点。
5. 对不确定内容，必须显式区分“已确认”“待验证”“无法判断”。
6. 输出应简洁、可执行、可复用，能直接转为测试点、测试单或回归清单。
7. 如果输入数据缺少关键字段，应在结果中明确提示缺口，不得自行补全业务事实。

## 5. 输出约束

1. 标题和章节顺序必须稳定。
2. 表格列名必须稳定。
3. 相同类型任务应保证输出结构一致。
4. 不允许输出与当前链路或本次变更无关的泛化废话。
5. 结论优先使用“影响点、风险点、验证点、观测点、待确认项”这类测试可执行语言。
6. 若存在优先级，统一使用 `P0 / P1 / P2`。
7. 若存在影响分类，优先使用：请求参数、响应字段、业务分支、状态流转、异常处理、依赖调用、持久化、鉴权、幂等、日志审计。
