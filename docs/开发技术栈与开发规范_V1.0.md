# 开发技术栈与开发规范（V1.0）

## 1. 文档目的

本文件用于明确项目统一技术栈与开发规范。后续功能开发、缺陷修复、性能优化、重构和运维脚本调整均需遵循本规范，避免技术分裂、环境不一致和质量失控问题。

## 2. 适用范围

本规范适用于以下内容：

1. 后端服务开发
2. 前端页面与组件开发
3. 数据库设计与变更
4. Redis 缓存设计与变更
5. 接口定义、发布和回归验证
6. 新增中间件、框架和基础组件引入

## 3. 技术栈基线（强制）

### 3.1 后端

- 语言与运行时：Java
- JDK 版本：JDK 21
- 框架：Spring Boot 2.7.6
- 构建工具：Maven
- ORM/持久层：MyBatis-Plus
- 数据库变更管理：Flyway

### 3.2 前端

- 框架：Vue
- 版本原则：必须与现有项目主版本保持一致，不允许 Vue 2 和 Vue 3 混用
- 包管理：沿用项目当前既有方案

### 3.3 数据层

- 关系型数据库：MySQL
- 缓存数据库：Redis

## 4. 版本与依赖管理规范

### 4.1 版本锁定原则

1. 生产环境与开发/测试环境必须使用同主版本技术栈。
2. Java 必须保持 JDK 1.8，不得私自升级到 JDK 11、17 或更高版本。
3. Spring Boot、Vue、MySQL、Redis 主版本升级必须走技术评审流程。

### 4.2 依赖引入原则

1. 后端新增依赖需说明用途、体积影响、安全风险和替代方案。
2. 前端新增 npm 包需评估维护活跃度、许可证风险和包体积影响。
3. 禁止引入重复能力库，例如多个日期库、多套路由方案、多套请求库并存。
4. 优先复用现有框架能力，避免为了局部问题引入重量级依赖。

### 4.3 兼容性原则

1. 所有新代码必须与 JDK 1.8 语法兼容。
2. 禁止使用仅高版本 Java 支持的语法特性和标准库 API。
3. 新增 SQL、缓存方案和前端组件必须与现网运行方式兼容。

## 5. 分层开发约束

### 5.1 后端（Java）

- Controller 层：仅做参数接收、参数校验、权限校验入口和响应封装，不承载复杂业务逻辑。
- Service 层：承载业务逻辑、事务边界和领域编排，不直接拼接 SQL。
- DAO/Mapper 层：专注数据访问，不混入业务规则。
- DTO/VO/DO：职责分离，禁止混用同一个对象跨层承载全部职责。
- 工具类：保持无状态、可复用，不侵入业务主流程。

### 5.2 前端（Vue）

- 页面与组件职责清晰，避免单组件过大。
- API 调用集中管理，不允许在多个页面散落重复请求逻辑。
- 统一状态管理方案，按现有项目标准执行。
- 公共组件优先复用，禁止复制粘贴式开发。
- 表单、列表、详情等通用交互保持一致性。

### 5.3 数据库（MySQL）

- 表结构变更必须通过 SQL 脚本或 Flyway 管理并记录版本。
- 禁止直接在生产库手工改表。
- 核心查询必须建立合理索引并经过执行计划验证。
- 涉及事务的业务必须显式定义事务边界。
- 变更脚本必须同时考虑默认值、历史数据兼容和回滚方案。

### 5.4 缓存（Redis）

- 所有缓存 key 必须采用统一命名规范：业务前缀:模块:主键。
- 必须设置合理 TTL，避免无界缓存。
- 缓存穿透、击穿、雪崩场景需有防护策略，如空值缓存、互斥锁、限流或降级。
- 缓存与数据库一致性方案需在设计文档中明确。
- 删除、更新和预热策略需要可追踪、可回滚。

## 6. 接口开发规范

1. 后端接口必须包含参数校验、异常处理和统一响应约定。
2. 公共接口变更必须说明向后兼容策略。
3. 接口字段命名、时间格式、分页格式、错误码风格必须与现有系统保持一致。
4. 涉及幂等、重试、限流、权限和审计的接口，设计文档中必须单独说明。
5. 新增接口应补充对应文档、调用示例和关键异常说明。

## 7. 代码规范与质量要求

1. 统一代码风格，提交前完成格式化与静态检查。
2. 新增功能必须包含必要注释，复杂业务逻辑必须说明意图和边界条件。
3. 禁止提交不可读的临时代码、调试代码、无意义日志和注释掉的大段废弃代码。
4. 复杂条件分支、缓存一致性、事务控制和容错逻辑必须可读、可测试。
5. 新增代码优先复用现有组件、工具类和基础设施能力。

## 8. 测试与验收要求

1. 后端新增功能至少补充必要的单元测试、集成测试或回归用例。
2. 前端必须处理接口失败、超时、空态和权限受限场景。
3. 涉及数据库脚本、Redis key 变更和公共接口变更时，必须覆盖关键路径验证。
4. 发布前必须完成基础回归与关键业务链路验证。
5. 高风险改动必须附带回滚预案和验证步骤。

## 9. 环境与安全规范

1. 开发环境、测试环境、生产环境配置分离。
2. 所有敏感信息必须通过配置中心或环境变量管理。
3. 禁止将密钥、账号、密码、令牌写入代码仓库。
4. 本地调试需使用与线上一致的关键中间件主版本，如 MySQL、Redis。
5. 日志中不得打印密码、令牌、身份证号、手机号等敏感信息明文。

## 10. 发布与变更要求

1. 涉及数据库变更时，必须附带回滚脚本。
2. 涉及 Redis key 结构变更时，需提供兼容或迁移方案。
3. 影响公共接口时，需说明向后兼容策略。
4. 发布前必须完成基础回归与关键路径验证。
5. 重大变更必须记录发布说明、风险点和回退方案。

## 11. 技术栈变更流程（必须执行）

当出现以下情况时，必须先完成技术评审，再实施变更：

- 升级 Java/JDK 版本
- 升级或切换 Vue 主版本
- 升级 MySQL、Redis 主版本
- 升级 Spring Boot 主次版本且影响兼容性
- 引入新的基础框架、中间件或核心基础库

评审输出至少包括：

1. 变更动机与收益
2. 兼容性影响评估
3. 迁移与回滚方案
4. 风险清单与应对措施
5. 分阶段实施计划

## 12. 执行说明

1. 本规范自发布之日起生效。
2. 新需求开发、重构与运维脚本调整均应遵循本规范。
3. 如与历史实现冲突，以“新增代码遵循规范、存量代码逐步治理”为原则推进。
4. 如需例外处理，必须在设计或评审结论中明确说明原因和临时边界。

- 后端框架：Spring Boot 2.7.6
- 前端：Vue
- 数据库：MySQL
- 缓存：Redis
- 数据变更管理：Flyway
- ORM：MyBatis-Plus
