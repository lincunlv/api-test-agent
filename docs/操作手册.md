# 接口测试智能体操作手册

## 1. 适用对象

本手册面向以下人员：

1. 测试工程师
2. 测试开发工程师
3. 需要联调接口测试智能体的开发人员

本手册重点说明“怎么操作”，不展开底层实现细节。

## 2. 你可以用它做什么

当前服务可以帮助你完成四类工作：

1. 梳理接口业务逻辑
2. 分析接口风险点
3. 生成测试用例草稿
4. 生成接口文档草稿

同时，服务还能：

1. 根据链路入口自动尝试补充源码上下文
2. 把每次任务的输入、状态、结果和事件日志落到独立目录
3. 通过任务查询接口查看执行结果

## 3. 使用前准备

### 3.1 启动服务

在项目根目录执行：

```bat
run.bat
```

或者：

```bash
mvn spring-boot:run
```

服务默认地址：

`http://localhost:8080`

### 3.2 准备接口链路数据

最少需要准备：

1. 接口名称
2. 接口路径
3. 请求方法

推荐额外准备：

1. 入口类名
2. 入口方法名
3. 包名
4. 参数个数

推荐示例：

```json
{
  "interface": {
    "name": "createOrder",
    "path": "/api/order/create",
    "method": "POST"
  },
  "entry": {
    "class_name": "OrderService",
    "method_name": "createOrder",
    "package_name": "com.demo.order.service",
    "parameter_count": 2
  }
}
```

## 4. 标准使用流程

推荐按下面顺序使用。

### 4.1 第一步：先做业务梳理

目标：

1. 理解接口做什么
2. 明确主流程和关键分支
3. 给后续风险分析和用例设计打基础

任务类型：

`A1`

示例请求：

```bash
curl -X POST http://localhost:8080/api/agent/tasks ^
  -H "Content-Type: application/json" ^
  -d "{\"taskType\":\"A1\",\"prompt\":\"请梳理该接口的业务流程和关键分支\",\"chainData\":{\"interface\":{\"name\":\"createOrder\",\"path\":\"/api/order/create\",\"method\":\"POST\"},\"entry\":{\"class_name\":\"OrderService\",\"method_name\":\"createOrder\",\"package_name\":\"com.demo.order.service\",\"parameter_count\":2}}}"
```

### 4.2 第二步：做风险分析

目标：

1. 找边界条件
2. 找异常路径
3. 找幂等、状态流转、空值等风险

任务类型：

`A2`

建议提示词：

`请分析该接口可能存在的风险点、异常分支和边界问题`

### 4.3 第三步：做测试用例生成

目标：

1. 形成测试用例草稿
2. 把主流程、边界和异常路径转成测试点

任务类型：

`A3`

建议提示词：

`请基于接口链路生成测试用例，覆盖主流程、异常分支和边界条件`

### 4.4 第四步：做接口文档生成

目标：

1. 形成接口说明草稿
2. 沉淀请求、响应和处理逻辑摘要

任务类型：

`A5`

建议提示词：

`请生成该接口的说明文档，包括请求、响应和处理逻辑摘要`

## 5. 如何查看结果

### 5.1 查询任务状态

当你创建任务后，会拿到 `taskId`。

通过下面接口查询：

```bash
curl http://localhost:8080/api/agent/tasks/{taskId}
```

重点关注返回中的：

1. `status`
2. `workspacePath`
3. `artifacts`
4. `sourceLookupApplied`
5. `message`
6. `lastError`

### 5.2 查看工作目录产物

每个任务都有独立工作目录。

常见文件包括：

1. `input.json`
2. `task.json`
3. `execution-events.jsonl`
4. `trace-summary.json`
5. `logic-analysis.md` / `risk-analysis.md` / `test-cases.md` / `api-doc.md`
6. `source-context.md`

### 5.3 推荐查看顺序

如果任务成功：

1. 先看主结果文件
2. 再看 `trace-summary.json`
3. 最后看 `execution-events.jsonl`

如果任务失败：

1. 先看 `task.json`
2. 再看 `trace-summary.json`
3. 最后看 `execution-events.jsonl`

## 6. 如何使用源码补查能力

### 6.1 自动方式

如果你在 `chainData.entry` 中提供：

1. `class_name`
2. `method_name`
3. `package_name`
4. `parameter_count`

服务会自动尝试补查源码。

命中后：

1. 返回结果中的 `sourceLookupApplied=true`
2. 工作目录中生成 `source-context.md`

### 6.2 手动方式

你也可以直接调用源码检索接口：

```bash
curl -X POST http://localhost:8080/api/agent/tools/get-method-source ^
  -H "Content-Type: application/json" ^
  -d "{\"className\":\"OrderService\",\"methodName\":\"createOrder\",\"packageName\":\"com.demo.order.service\",\"parameterCount\":2}"
```

适合在以下场景使用：

1. 想先验证方法能否定位到
2. 某个任务未命中自动补查时，手动排查原因
3. 需要单独查看关键方法实现时

## 7. 如何使用 renderer 模式

### 7.1 `template`

用途：

1. 使用默认模板输出
2. 验证工作流和文件落盘

### 7.2 `mock-model`

用途：

1. 联调前端、平台、任务流程
2. 不接真实模型，直接指定返回内容

示例：

```json
{
  "options": {
    "rendererMode": "mock-model",
    "mockOutput": "# 自定义输出\n\n这是模拟模型结果"
  }
}
```

### 7.3 `mock-failure`

用途：

1. 验证失败场景
2. 验证 500 响应和失败归档

示例：

```json
{
  "options": {
    "rendererMode": "mock-failure",
    "mockFailureMessage": "forced renderer failure"
  }
}
```

## 8. 常见问题

### 8.1 为什么结果内容像模板，不像真实分析

因为当前版本还没有接入真实 LLM，只是把流程、归档、Skill 和补源码能力打通了。

### 8.2 为什么任务成功了，但内容仍然是“待补充”

这是当前骨架版本的正常现象。`PARTIAL_SUCCESS` 代表工作流成功、结果已生成，但分析内容仍是模板草稿。

### 8.3 为什么没有生成 `source-context.md`

通常有三种原因：

1. `chainData.entry` 不完整
2. 类名/方法名/包名/参数个数不匹配
3. 本地源码路径下没有找到对应 Java 文件

### 8.4 如果任务失败，先看什么

建议顺序：

1. `task.json`
2. `trace-summary.json`
3. `execution-events.jsonl`

## 9. 适合当前版本的使用方式

当前最适合这样使用：

1. 用于平台联调
2. 用于验证接口测试智能体工作流
3. 用于输出测试资产骨架
4. 用于演示如何把链路数据转成结构化结果

如果你要把它真正用于生产级接口测试智能分析，下一步应继续接入真实模型能力。
