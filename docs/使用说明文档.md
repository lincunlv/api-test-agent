# 接口测试智能体使用说明文档

## 1. 文档目的

本文档面向当前服务的使用者，说明以下内容：

1. 当前服务已经具备哪些能力。
2. 如何启动和调用这些能力。
3. 如何基于这些能力完成接口测试分析、风险识别、用例设计和文档产出。
4. 当前版本的边界和适用方式。

## 2. 服务定位

当前版本的接口测试智能体，是一个“可运行的接口测试分析骨架服务”。

它已经具备完整的任务编排、Skill 路由、结果落盘、事件归档、源码补查、中间件管道和测试基础，但尚未接入真实 LLM/Agent Runtime。因此，它当前更适合用于以下场景：

1. 验证接口测试智能体的整体工作流。
2. 作为后续接入真实模型前的工程底座。
3. 演示如何把接口链路数据转为结构化测试资产。
4. 联调平台侧的任务创建、状态查询、产物读取和归档逻辑。

如果你希望当前版本直接输出高质量、真实推理的业务分析内容，需要在现有 renderer 接缝上继续接入真实模型。

## 3. 当前能力概览

### 3.1 已实现能力

当前服务已经具备以下能力：

1. 任务创建接口：可提交接口链路数据和分析请求。
2. 任务查询接口：可查看任务状态、产物、事件计数和错误信息。
3. Skill 路由：支持 A1、A2、A3、A4、A5 五类任务。
4. 工作空间落盘：任务执行时自动创建独立目录并产出文件。
5. 文件化状态流转：支持 `PENDING -> RUNNING -> PARTIAL_SUCCESS/FAILED`。
6. 事件归档：生成 `execution-events.jsonl` 和 `trace-summary.json`。
7. 方法源码补查：支持按类名、方法名、包名、参数个数检索 Java 方法源码。
8. 自动补源码上下文：任务执行时可根据链路 `entry` 自动尝试补查源码。
9. 分析中间件管道：当前包含 Guardrail、ContextEditing、Observability、Retry、ErrorBoundary。
10. 可配置 renderer 接缝：支持 `template`、`mock-model`、`mock-failure` 等模式，用于后续模型接入和测试。

### 3.2 当前不具备的能力

当前版本暂不具备以下能力：

1. 真实大模型推理。
2. 真正基于源码语义的复杂业务分析。
3. 自动执行 HTTP 接口测试。
4. 自动断言接口响应和自动生成运行脚本。
5. 数据库、Redis、外部依赖的真实联动验证。

## 4. 五类任务说明

系统当前支持五类任务，通过 `taskType` 指定：

### 4.1 A1 业务逻辑梳理

用途：

1. 梳理接口业务目标。
2. 明确主流程和关键分支。
3. 输出 Mermaid 流程图骨架。

适合在接口测试准备阶段，用来先建立“接口到底做了什么”的统一理解。

当前 A1 输出会额外强调测试视角摘要、关键分支验证点、外部依赖失败影响和测试关注点，适合为后续回归范围收敛提供输入。

### 4.2 A2 Bug 风险分析

用途：

1. 识别风险点、边界问题和异常场景。
2. 形成风险清单表格。

适合在测试设计阶段，用来先找风险，再决定测什么。

当前 A2 输出会给出风险结论摘要、关键观测点和优先验证建议，更适合快速安排风险优先级。

### 4.3 A3 测试用例生成

用途：

1. 输出结构化测试用例表。
2. 形成主流程、边界和异常测试点草稿。

适合把链路数据转成测试资产。

当前 A3 输出除测试用例表外，还会补充测试数据/输入、核心观测点和执行顺序建议，更利于直接落成测试单。

### 4.4 A4 Diff 场景级测试脚本生成

用途：

1. 基于 git diff 识别本次改动影响了哪些接口、能力和业务场景。
2. 生成基于本次改动的场景级测试脚本草稿。
3. 补充跨接口旧能力的回归脚本建议和执行顺序。

适合在代码变更、提测前自检、回归范围评估阶段使用。

当前 A4 输出会重点包含：变更摘要、测试脚本生成判断、关联接口识别、受影响场景识别、增量测试脚本、回归补充脚本和执行顺序建议。

### 4.5 A5 API 文档生成

用途：

1. 生成接口说明文档。
2. 补齐请求、响应和处理逻辑摘要。

适合在测试和开发协作时，先形成可阅读的接口说明草稿。

当前 A5 输出会补充测试与联调提示，方便测试在联调阶段快速确定前置数据和观测点。

## 5. 启动方式

### 5.1 前置条件

需要本地具备：

1. JDK 1.8
2. Maven

### 5.2 启动命令

Windows 下可直接执行：

```bat
run.bat
```

或者使用 Maven：

```bash
mvn spring-boot:run
```

默认端口：

`8080`

## 6. 核心接口说明

### 6.1 创建分析任务

接口：

`POST /api/agent/tasks`

请求体结构：

```json
{
  "taskType": "A3",
  "prompt": "请生成测试用例",
  "chainData": {
    "interface": {
      "name": "createOrder",
      "path": "/api/order/create",
      "method": "POST"
    },
    "entry": {
      "class_name": "OrderService",
      "method_name": "createOrder",
      "package_name": "com.demo.order.service",
      "parameter_count": 2
    }
  },
  "options": {
    "rendererMode": "template"
  }
}
```

字段说明：

1. `taskType`：任务类型，支持 `A1`、`A2`、`A3`、`A4`、`A5`。
2. `prompt`：用户意图描述。
3. `chainData`：接口链路数据。
4. `options`：执行选项。

常用 `options` 补充项：

1. `rendererMode`：切换模板、mock 成功、mock 失败等渲染模式。
2. `methodSourceQuery`：手动指定源码补查条件。
3. `gitDiffQuery`：自动采集 git diff，并供 A4 生成增量测试与回归方案使用。

返回结果中重点关注：

1. `taskId`
2. `status`
3. `workspacePath`
4. `artifacts`
5. `sourceLookupApplied`

### 6.2 查询任务状态

接口：

`GET /api/agent/tasks/{taskId}`

用途：

1. 查询任务当前状态。
2. 查看是否命中源码补查。
3. 查看已产出的文件。
4. 查看错误信息和事件计数。

### 6.3 查询方法源码

接口：

`POST /api/agent/tools/get-method-source`

请求体示例：

```json
{
  "className": "AgentTaskService",
  "methodName": "createTask",
  "packageName": "com.apitestagent.service",
  "parameterCount": 1
}
```

适用场景：

1. 任务执行前先验证源码检索是否命中。
2. 分析某个接口关键方法时补充上下文。
3. 联调平台侧源码查看能力。

### 6.4 查询 Git Diff

接口：

`POST /api/agent/tools/get-git-diff`

适用场景：

1. A4 任务执行前先验证本次变更是否被正确采集。
2. 识别 `changedFiles`、`changedMethods` 和 `relatedInterfaces` 是否符合预期。
3. 用于提测前做增量测试范围评估。

## 7. options 可选能力说明

### 7.1 `rendererMode`

当前支持的模式：

1. `template`：默认模式，输出模板化结构内容。
2. `mock-model`：返回 `mockOutput` 指定的内容，用于联调或测试。
3. `mock-failure`：主动抛出错误，用于验证 500 契约和失败归档。

### 7.2 `mockOutput`

仅在 `rendererMode=mock-model` 时生效。

示例：

```json
{
  "rendererMode": "mock-model",
  "mockOutput": "# 自定义输出\n\n这里是模拟模型返回内容"
}
```

### 7.3 `mockFailureMessage`

仅在 `rendererMode=mock-failure` 时生效。

示例：

```json
{
  "rendererMode": "mock-failure",
  "mockFailureMessage": "forced renderer failure"
}
```

## 8. 任务产物说明

任务执行后，会在 `workspace/agent_files/{skill}/{taskId}/` 下生成文件。

常见产物包括：

1. `input.json`：原始请求快照。
2. `task.json`：任务状态快照。
3. `execution-events.jsonl`：阶段事件流水。
4. `trace-summary.json`：任务摘要。
5. `logic-analysis.md` / `risk-analysis.md` / `test-cases.md` / `diff-test-cases.md` / `api-doc.md`：主结果文件。
6. `source-context.md`：自动补充的源码上下文。
7. `git-diff.patch`：自动采集的代码变更内容。
8. `diff-impact.json`：A4 使用的结构化变更影响摘要。

### 8.1 如何看 `execution-events.jsonl`

关键阶段通常包括：

1. `SUBMITTED`
2. `STATUS_CHANGED`
3. `INPUT_PERSISTED`
4. `SKILL_LOADED`
5. `SOURCE_LOOKUP_SKIPPED` 或 `SOURCE_LOOKUP_FOUND`
6. `ARTIFACT_WRITTEN`
7. `COMPLETED` 或 `FAILED`

这个文件适合用来排查任务为什么成功、为什么失败、卡在哪一步。

### 8.2 如何看 `trace-summary.json`

建议优先看以下字段：

1. `status`
2. `artifacts`
3. `executionEventCount`
4. `sourceLookupApplied`
5. `message`
6. `lastError`

## 9. 如何用当前服务实现接口测试

虽然当前版本还没有接入真实模型，但已经可以用于“接口测试流程化”和“测试资产骨架生成”。推荐使用方式如下。

### 9.1 第一步：准备接口链路数据

至少准备：

1. 接口名称
2. 接口路径
3. 请求方法
4. 入口方法信息

推荐把 `entry` 一起带上，这样系统会自动尝试源码补查。

### 9.2 第二步：先做 A1 业务梳理

目标：

1. 建立接口测试对象的业务理解。
2. 确认主流程和关键分支。
3. 为后续风险分析和用例设计建立输入基线。

建议请求：

```json
{
  "taskType": "A1",
  "prompt": "请梳理该接口的业务流程和关键分支",
  "chainData": {
    "interface": {
      "name": "createOrder",
      "path": "/api/order/create",
      "method": "POST"
    },
    "entry": {
      "class_name": "OrderService",
      "method_name": "createOrder",
      "package_name": "com.demo.order.service",
      "parameter_count": 2
    }
  }
}
```

### 9.3 第三步：做 A2 风险分析

目标：

1. 找出边界、异常、幂等、状态流转等风险点。
2. 形成测试关注清单。

适合用于确定哪些场景必须重点测。

### 9.4 第四步：做 A3 测试用例生成

目标：

1. 基于前面识别的主流程和风险点，形成用例草稿。
2. 输出统一表格，便于后续整理成测试案例。

当前版本生成的是结构化骨架，更适合作为测试用例模板，而不是最终版测试资产。

### 9.5 第五步：如有代码改动，插入 A4 增量回归分析

目标：

1. 判断这次改动影响了哪些接口和旧能力。
2. 输出增量测试点、重点回归项和执行顺序建议。

适合在拿到 git diff 后，快速收敛回归范围。

建议请求：

```json
{
  "taskType": "A4",
  "prompt": "请根据本次 git diff 识别直接影响接口、间接受影响接口，并生成接口测试脚本与回归补充脚本建议",
  "chainData": {
    "interface": {
      "name": "getGitDiff",
      "path": "/api/agent/tools/get-git-diff",
      "method": "POST"
    }
  },
  "options": {
    "gitDiffQuery": {
      "repositoryPath": "g:/api-test-agent-service/api-test-agent",
      "cached": true,
      "pathspecs": [
        "src/main/java",
        "src/test/java",
        "skills/interface-chain-analyzer"
      ]
    }
  }
}
```

### 9.6 第六步：做 A5 文档生成

目标：

1. 形成接口说明草稿。
2. 帮助测试、开发、产品统一接口理解。

### 9.6 推荐的串联流程

建议顺序如下：

1. A1：先梳理业务流程。
2. A2：再做风险分析。
3. A3：据此形成测试用例草稿。
4. A4：如涉及代码改动，再补增量测试和回归分析。
5. A5：最后输出接口文档草稿。

这套顺序的意义是：

1. 先理解接口做什么。
2. 再识别哪里容易出错。
3. 再决定该怎么测。
4. 若有变更，再确认哪些能力必须回归。
5. 最后把理解沉淀成文档。

## 10. 推荐实践

### 10.1 联调阶段推荐使用 `mock-model`

如果你现在主要是打通平台、页面、文件浏览和任务编排，建议使用：

1. `rendererMode=mock-model`
2. `mockOutput=自定义内容`

这样可以稳定验证全链路，不依赖真实模型。

### 10.2 开发阶段推荐保留 `entry`

只要链路里能提供入口类和方法，系统就能尝试自动补源码上下文。即使当前还不是真实语义分析，这个能力也能帮助你提前建立“按需查源码”的工作流。

### 10.3 排查问题优先看归档文件

排查顺序建议：

1. 先看 `task.json`
2. 再看 `trace-summary.json`
3. 最后看 `execution-events.jsonl`

## 11. 异常响应说明

当前接口异常响应约定如下：

1. `VALIDATION_ERROR`：请求参数缺失或格式非法。
2. `BAD_REQUEST`：业务参数不合法，例如不支持的 `taskType`。
3. `NOT_FOUND`：请求的任务不存在。
4. `INTERNAL_ERROR`：内部运行时异常，例如 `mock-failure` 模式主动注入的故障。

## 12. 当前版本边界

使用当前版本时，需要明确以下边界：

1. 输出内容默认仍是模板化骨架，不是实际模型推理结果。
2. 当前更适合验证工作流、任务编排、落盘和接口契约。
3. 如果要把它真正用于接口测试智能分析，需要继续接真实模型适配器。
4. `get-method-source` 当前是文件级和方法签名级匹配，不是 AST 级语义解析。

## 13. 下一步建议

如果要把当前服务用于更真实的接口测试场景，建议按这个顺序继续推进：

1. 接入真实 LLM/Agent Runtime，替换当前模板 renderer。
2. 继续增强链路输入结构，补充参数、分支、异常和依赖信息。
3. 把 `get-method-source` 升级为 AST 或语义级检索。
4. 增加真正的接口执行、断言和测试报告能力。
