# 接口测试智能体开发计划

## 1. 计划目标

本计划用于将“接口测试智能体”从需求设计推进到可上线的生产版本，覆盖研发、测试、运维和验收。计划以“可执行”为原则，拆分为明确阶段、交付物、责任角色和完成标准。

## 2. 开发策略

### 2.1 总体策略

1. 先闭环，再扩展。
2. 先标准化 Skills，再开发 Agent 编排。
3. 先保障稳定性，再扩展模型能力。
4. 所有核心能力都必须有样例、回归和黄金输出。

### 2.2 交付路线

```mermaid
flowchart LR
    A[阶段0 方案澄清] --> B[阶段1 基础骨架]
    B --> C[阶段2 Skills与Agent闭环]
    C --> D[阶段3 中间件与稳定性]
    D --> E[阶段4 质量与观测]
    E --> F[阶段5 试点上线]
```

## 3. 团队角色建议

### 3.1 人员配置

1. 产品经理 1 人
2. 后端/Agent 开发 2 人
3. 测试开发 1 人
4. 平台/运维 1 人（可兼职）

### 3.2 责任划分

- 产品经理：需求收敛、Skill 输出规范、验收口径
- Agent 开发：执行链路、工具、文件路由、中间件
- 测试开发：回归集、黄金输出、自动化验收
- 平台/运维：部署、监控、告警、权限隔离

## 4. 阶段计划

以下排期按 6 周执行估算，可根据团队规模压缩或拉长。

## 8. 阶段 3：工具能力与中间件稳定性

### 8.1 时间

- 预计 8 个工作日

### 8.2 开发任务

1. 实现 get_method_source 工具。
2. 接入方法级源码索引或检索服务。
3. 实现 RequestContextMiddleware。
4. 实现 ObservabilityMiddleware。
5. 实现 RetryMiddleware。
6. 实现 GuardrailMiddleware。
7. 实现 ContextEditingMiddleware。
8. 实现 ErrorBoundaryMiddleware。

### 8.3 ContextEditing 实施细则

1. 加入 token 估算模块。
2. 设置 100000 token 触发阈值。
3. 设置保留最近 5 条工具结果。
4. 设置至少清理 20000 token。
5. 排除写文件相关工具记录。
6. 补充清理前后日志和指标上报。

### 8.4 测试任务

1. 构造长链路和多轮对话超长场景。
2. 验证上下文清理后任务可继续执行。
3. 验证工具失败重试、不可恢复失败降级输出。
4. 验证会话隔离与目录权限。

### 8.5 交付物

- 自定义工具能力
- 6 层中间件实现
- 稳定性测试报告

### 8.6 完成标准

- 长链路分析场景无明显 token 超限阻塞
- 工具可按需补充方法源码

## 13. 风险管理计划

### 13.1 风险项与应对

1. 链路数据字段不稳定。
应对：先定义最小字段集和兼容转换层。

2. 输出质量不一致。
应对：建立黄金输出、格式校验器和 Skill 回归评审。

3. 长上下文导致失败。
应对：优先实现 ContextEditingMiddleware，并在阶段 3 完成压测。

4. 源码检索召回不足。
应对：增加 file_hint、类索引和缓存策略。

5. 生产观测缺失。
应对：阶段 1 即接入结构化日志，阶段 4 完成监控大盘。

## 14. 度量指标

### 14.1 研发过程指标

- 每阶段计划完成率
- 单元测试覆盖率
- 回归通过率
- 缺陷逃逸率

### 14.2 产品效果指标

- 任务成功率
- 平均耗时
- 结构化输出一致性
- 风险命中率
- 测试用例采纳率
- 文档可直接复用率

## 15. 发布门禁

系统进入试点上线前必须满足：

1. P0 项全部完成。
2. 四类 Skill 各自至少有 10 条通过样例。
3. 长链路压测通过。
4. 监控和错误日志可追踪。
5. 输出结构校验器默认开启。
